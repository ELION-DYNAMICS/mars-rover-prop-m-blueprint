# Calibration Framework

This directory contains the formal mobility calibration pipeline for the
Mars Rover Prop-M Blueprint.

It exists to turn datasets into defensible physical parameters.

Calibration is parameter identification under controlled conditions.

---

# 1. Purpose

The calibration framework identifies and validates:

- Slip behavior (E01)
- Traction envelope (E02)
- Static sinkage parameters (E03)
- Incline force balance consistency (E04)

Outputs from calibration feed:

- terramechanics_defaults.yaml
- estimator_defaults.yaml
- control_defaults.yaml

No parameter may be promoted to runtime configuration without:

1. Dataset reference
2. Reproducible script
3. Acceptance check
4. Regression confirmation

---

# 2. Directory Structure


analysis/calibration/
data/
raw/
processed/
manifests/
configs/
experiments/
outputs/
reports/
tools/


### data/
Contains dataset manifests and optional derived artifacts.

Authoritative raw datasets live in:

`/datasets/<run_id>/`

This directory must not duplicate raw logs unless explicitly required.

---

### configs/
Calibration baseline parameters.

These files are *inputs* to fitting and *outputs* of refinement.

They are not runtime configs.
Runtime promotion happens in `/configs/` only after review.

---

### experiments/
Each experiment is isolated and must contain:

- spec.md
- fit.py
- plot.py (optional)
- documented outputs

Experiments:

- E01 – Straight-line slip
- E02 – Torque ramp traction
- E03 – Static sinkage
- E04 – Incline climb validation

Order is mandatory.
Skipping experiments invalidates conclusions.

---

### outputs/
Run-specific artifacts:


outputs/<run_id>/E0X/
summary.json
plots
updated parameter YAML


These are generated by scripts.

---

### reports/
Human-reviewed summaries and consolidated calibration reports.

This directory must not contain autogenerated intermediate files.

---

### tools/
Shared deterministic utilities:

- Dataset loader
- Fit utilities
- Metrics
- Plotting

No experiment may implement hidden numerical shortcuts outside tools/.

---

# 3. Calibration Discipline

Calibration must:

- Use deterministic seeds
- Use bounded optimization
- Enforce physically plausible parameter limits
- Fail loudly on inconsistent data
- Produce traceable artifacts

Visual plausibility is not acceptance.

Acceptance requires quantified metrics.

---

# 4. Reproducibility Standard

Every calibration result must be reproducible with:

- Same run_id
- Same commit hash
- Same parameter initialization
- Same seed

The following must match within tolerance:

- Fitted parameters
- Summary statistics
- Acceptance result

If two runs produce materially different outputs,
investigate before promotion.

---

# 5. Data Integrity Rules

Calibration scripts must:

- Read from dataset_index.json
- Validate dataset paths exist
- Refuse to run on unusable datasets
- Never overwrite raw data

If a dataset is incomplete,
mark it unusable in the manifest.

---

# 6. Parameter Promotion Policy

After successful calibration:

1. Export updated YAML into `outputs/`
2. Review summary report
3. Confirm regression scenarios pass
4. Manually promote parameters to `/configs/`

No automatic promotion.

---

# 7. Engineering Philosophy

The purpose of calibration is to enforce reality on the model.

If a model requires unrealistic parameters to match data,
the model is wrong.

If data produces unstable fits,
the experiment is wrong.

The objective is internal physical consistency,
not curve matching.

---

# 8. Long-Term Objective

This framework should eventually support:

- Automated nightly parameter validation
- Regression comparison across soil profiles
- Parameter uncertainty estimation
- Confidence interval reporting
- Dataset-backed model evolution

Calibration is the scientific backbone of mobility.

Without it, the rover is a simulator.
With it, the rover becomes an engineering system.
